# ADR-0001: Mirror Architecture and Segmentation Direction

- Status: Accepted
- Date: 2026-04-12
- Owners: juan.software experiments

## Context

A new experiment (`mirror`) requires persistent live reflection with two modes:

1. full-frame liquid/chrome mirror
2. particle mirror generated by the human subject only

The system must run browser-side, low-latency, and reinforce long-term reuse across experiments.

## Decision

Adopt a modular creative architecture with explicit stages:

`camera -> segmentation -> adapters -> particle simulation -> renderer`

Implement shared infrastructure under `src/lib/creative/*` and keep Mirror-specific render behavior under `src/lib/mirror/*`.

Use browser-side person segmentation through `@mediapipe/tasks-vision`.

## Why This Architecture

- keeps concerns isolated and composable
- avoids one-off coupling between camera and rendering
- supports future experiments reusing camera/segmentation/particle modules
- preserves Resonance lineage (imperative controller + explicit frame order)

## Why Browser-Side Person Segmentation

- satisfies requirement: particle mode should represent people only
- no server round-trips, preserving latency and privacy
- mature runtime path for realtime browser inference
- lighter integration footprint than a broader tfjs graph stack for MVP

## Why OCR / Generative AI Are Not Used

- task is body presence segmentation, not semantic text or generation
- OCR and cloud AI add latency/cost/privacy complexity without product benefit
- local segmentation directly matches the interaction requirement

## Why Liquid Mode Uses Full Camera Frame

- mirror behavior requires full-scene continuity
- avoids “masked-only” feeling in liquid mode
- preserves immediate real-world reflectivity and spatial coherence

## Tradeoffs

- MediaPipe model is fetched at runtime from official remote asset URLs
- Canvas2D liquid effect is stylized and maintainable, not physically-based rendering
- segmentation quality can vary with lighting and webcam quality

## Performance and Maintainability Balance

- quality tiers tune segmentation cadence and particle density
- segmentation loop is decoupled from render loop
- fallback to liquid mode keeps the route functional if segmentation fails

## Consequences

Positive:

- establishes a durable creative-system baseline for future experiments
- improves system-level reuse while keeping experiments distinct

Negative:

- introduces one additional dependency and runtime model fetch path
- requires ongoing compatibility checks for model URLs / browser delegates

## Follow-ups

- optionally self-host segmentation model assets
- add runtime telemetry for adaptive quality tuning
- evaluate optional WebGL renderer behind same mode interfaces
